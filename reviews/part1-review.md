## ❌ AI 생성 티가 나는 문장 (4개)

### 1. [Slide 2-1] 도입 판단 5단계 script
**원문**: "GraphRAG를 도입할지 말지, 5단계로 판단하세요."
**문제**: 교과서적 지시문. "5단계로 판단하세요"는 근거 없는 명령.
**수정 제안**: "우리 팀에서 실제로 GraphRAG 도입을 검토할 때 거쳤던 단계입니다. 처음엔 3단계였는데, 실패하고 나서 5단계로 늘었어요."

### 2. [Slide 2-2] 3가지 경험 유형 script
**원문**: "GraphRAG를 경험하는 방법은 크게 3가지입니다. 우리는 Type 2로 갑니다."
**문제**: "크게 3가지입니다" — 전형적 AI 나열 패턴. 왜 Type 2인지 근거가 없음.
**수정 제안**: "GraphRAG를 경험하는 방식이 여러 개인데, 실무에서 직접 부딪혀보니 이 3가지로 나뉘더라고요. 우리는 Type 2를 선택했는데, 이유가 있습니다 — Type 1(MS GraphRAG)은 커뮤니티 요약 방식이라 Cypher를 안 써요. 근데 실무에서 디버깅하려면 Cypher를 직접 볼 수 있어야 합니다."

### 3. [Slide 3-1] 온톨로지 = 합의의 도구 script
**원문**: "온톨로지는 '이 도메인에서 무엇을 엔티티로 보고, 무엇을 관계로 볼 것인가'에 대한 합의입니다."
**문제**: 정의문 자체는 정확하지만, "합의"가 왜 어려운지 구체적 사례가 없음.
**수정 제안**: "온톨로지가 뭐냐 — 한마디로 '우리 팀이 합의한 데이터 사전'입니다. 근데 이게 쉬워 보이지만, 실제로 해보면 팀원 3명이 각각 다른 걸 엔티티로 뽑아요. 예를 들어 '열압착 공정'이 하나의 엔티티인지, '열'과 '압착'이 각각 엔티티인지 — 이런 거 합의하는 데 보통 2-3일 걸립니다."

### 4. [Slide 5-1] "왜 Neo4j인가" script
**원문**: "우리가 Neo4j를 쓰는 이유는 딱 하나입니다."
**문제**: "딱 하나"라고 하고 4가지를 나열. 일관성 없음.
**수정 제안**: "Neo4j를 선택한 이유를 물으면 저는 3가지로 답합니다. 첫째, Cypher 문법이 SQL 닮아서 러닝커브 낮음. 둘째, LangChain이 Neo4j를 1급 시민으로 지원. 셋째, 무료 Docker 이미지로 5분 만에 시작 가능. 나머지 GDBMS 비교는 Part 7에서 합니다."

## ❌ 기술적 부정확 (3개)

### 1. [Slide 6-1] Docker Compose version 필드
**문제**: `version: '3.8'` — Docker Compose V2에서는 `version` 필드가 deprecated됨. 최신 Docker Compose는 이 필드를 무시함.
**수정**:
```yaml
# docker-compose.yml
services:
  neo4j:
    image: neo4j:5-community
    ports:
      - "7474:7474"
      - "7687:7687"
    environment:
      NEO4J_AUTH: neo4j/password123
    volumes:
      - ./data:/data
```
- `version` 필드 제거
- `neo4j:5.15-community` → `neo4j:5-community` (항상 5.x 최신)

### 2. [Slide 1-4] vector_search 의사코드
**문제**: `vector_search(query, top_k=5)` — 존재하지 않는 함수. 학습자가 실행할 수 없음.
**수정**: 실제 LangChain 또는 OpenAI Embeddings 코드로 교체하거나, 명시적으로 "의사코드"임을 표시.
```python
# ⚠️ 개념 설명용 의사코드 (실행 불가)
query = "삼성전자에 투자한 기관이 투자한 다른 반도체 기업은?"
results = vector_store.similarity_search(query, k=5)
# → 청크 단위 검색이므로 multi-hop 관계를 연결할 수 없음
```

### 3. [Slide 4-1] 6레이어 프레임워크 불일치
**문제**: 가이드 v3.0에서는 `L1 Strategy → L2 Data → L3 Infra → L4 Processing → L5 Deployment`인데, 콘텐츠에서는 `L1 Infra → L2 Ontology → L3 Extraction → L4 Retrieval → L5 Advanced`로 다름. 레이어 이름과 순서가 다른 버전.
**수정**: 가이드 v3.0 기준으로 통일하거나, 콘텐츠용 프레임워크를 "학습 로드맵"으로 리네이밍하여 혼동 방지.

## ❌ 깊이 부족 (5개)

### 1. [Section 1 전체] 제조 도메인 연결 부재
**현재**: 벡터 RAG 한계를 금융/투자 예시("삼성전자에 투자한 기관...")로만 설명.
**문제**: 이 과정의 케이스 스터디는 제조 도메인(브레이크 패드)인데, Part 1에서 제조 예시가 단 하나도 없음.
**추가 필요**: 제조 도메인 Multi-hop 예시를 병행 제시.
```
금융: "삼성전자에 투자한 기관이 투자한 다른 반도체 기업은?"
제조: "접착 박리 결함이 발생한 공정에서 사용된 설비의 최근 정비 이력은?"
       (결함 → 공정 → 설비 → 정비이력 = 3-hop)
```

### 2. [Slide 2-1] 도입 판단 5단계 — 각 단계 설명 부재
**현재**: 5단계를 다이어그램으로 나열만 함.
**문제**: 각 단계가 무엇을 의미하고, 어떻게 실행하는지 설명이 없음.
**추가 필요**: 최소한 Step 1(초기 인터뷰)에 대해 "어떤 질문을 던져야 하는지" 예시 2-3개.

### 3. [Slide 3-2] "BFS/DFS (빠름)" 과도한 단순화
**현재**: LPG의 Multi-hop 성능을 "BFS/DFS (빠름)"으로 한 단어 정리.
**문제**: 왜 빠른지 설명 없음. RDB는 N-1 JOIN이 필요하지만 Graph DB는 인접 노드 탐색이 O(degree)인 이유.
**추가 필요**: "SQL 3-hop = JOIN 3번 (테이블 풀스캔 가능) vs Cypher 3-hop = 인접 노드 3번 탐색 (인덱스 불필요)" 정도의 직관적 비교.

### 4. [Section 5] "왜 Neo4j인가" 단 1슬라이드
**현재**: 5분 분량에 슬라이드 1개.
**문제**: GDBMS 선택이 Part 7로 미뤄졌다는 것은 좋지만, "왜 이 과정에서 Neo4j를 쓰는지"에 대한 설득이 부족.
**추가 필요**: DB-engines 그래프 DB 랭킹 1위라는 사실, LangChain neo4j 통합 코드 한 줄 예시 정도.

### 5. [Section 6] 실습 결과 검증 포인트 없음
**현재**: CREATE → MATCH → 시각화 순서로 진행.
**문제**: "이 결과가 나와야 정상"이라는 체크포인트가 없음. 학습자가 실수했을 때 알 방법이 없음.
**추가 필요**: 각 실습 단계 후 `MATCH (n) RETURN count(n)` 같은 검증 쿼리 + "노드 5개, 관계 3개가 보여야 정상" 체크리스트.

### 3. 벡터 RAG 실제 실패 before/after
실제 LangChain 코드로 벡터 RAG를 실행하고, 틀린 답이 나오는 것을 보여준 후, 같은 질문을 Cypher로 정확하게 답하는 before/after.

### 4. 실습 완료 체크리스트
```
✅ Neo4j Docker 컨테이너가 실행 중이다 (7474 포트 접속 확인)
✅ 노드 5개가 생성되었다 (MATCH (n) RETURN count(n) = 5)
✅ 관계 3개가 생성되었다 (MATCH ()-[r]->() RETURN count(r) = 3)
✅ Multi-hop 쿼리가 결과를 반환한다
✅ Neo4j Browser에서 그래프가 시각화된다
```

### 5. "왜 이 순서인가" 설명
Part 1이 전체 커리큘럼에서 어떤 역할을 하는지, Part 2로 넘어가기 전에 알아야 할 것이 무엇인지.

### 6. Cypher 치트시트
Section 6 실습에서 사용할 기본 Cypher 문법 요약 카드:
- CREATE (node:Label {prop: value})
- MATCH (n:Label) WHERE n.prop = value RETURN n
- MATCH (a)-[:REL]->(b) RETURN a, b
