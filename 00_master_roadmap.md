## 2. 핵심 전략: 왜 선두를 잡을 수 있는가

### 2-1. 시장 분석

```
현재 한국 시장 상황:

벡터 RAG ────── 레드오션 🔴
  ├── 수많은 튜토리얼, 블로그, 강의
  ├── 진입장벽 낮음
  └── 차별화 어려움

GraphRAG ────── 블루오션 🔵
  ├── 체계적 가이드 부재
  ├── 진입장벽 높음 (온톨로지 + 그래프DB + LLM)
  ├── 도메인별 유스케이스 거의 없음
  └── ⭐ 선점 효과 극대화 가능

Root Bricks 포지셔닝:
  "GraphRAG를 배우고 싶으면 여기로 오면 된다"
  "도메인별 실무 유스케이스가 다 있다"
  "미니 데모부터 서비스급까지 단계별로 체험 가능"
```

### 2-2. 경쟁 우위 30개 항목 (커리큘럼 v3.0 기반)

커리큘럼 v3.0에서 확보한 차별화 항목이 그대로 유스케이스에 녹아든다:

| 카테고리 | 우리만의 차별점 | 유스케이스 반영 |
|----------|---------------|---------------|
| 도입 판단 | GraphRAG 도입 판단 5단계 프로세스 | 각 도메인 개요에 포함 |
| 관계 설계 | Relationship Prefix 9가지 | 온톨로지 스키마에 반영 |
| 관계 추출 | Meta-Dictionary (관계 패턴 JSON) | 0단계부터 제공 |
| 체감 학습 | 수작업 → LLM 자동화 비교 | 0→1단계 전환에서 체감 |
| 프롬프트 | "구체화 > 일반화" 원칙 | 엔티티 추출 프롬프트에 적용 |
| 검색 | 기본 Text2Cypher → Agent 단계 분리 | 0단계(기본) → 1단계(Agent) |
| 평가 | Multi-hop + Common Knowledge 메트릭 | 2단계부터 RAGAS 포함 |
| 아키텍처 | GDBMS 선정 3기준 + 벤치마크 | 3단계 실무 가이드 |

## 3. 유스케이스 4단계 구조 (v2.0 — 0단계 추가)

### 3-1. 설계 철학

```
커리큘럼 v3.0 핵심 원칙:
  "처음부터 서비스급으로 하면 다들 모른다"
  "첫 성공 경험이 가장 중요하다"
  "어려운 것을 쉽게 설명한다"

          ↓ 유스케이스에 적용

  0단계: 미니 데모  → "아, 이런 거구나!" (5분 만에 동작)
  1단계: 교육용    → "직접 해보니 다르구나" (수작업 체감)
  2단계: 프로토    → "실무에선 이렇게 쓰는구나" (자동화 체감)
  3단계: 서비스급  → "이걸로 사업할 수 있겠다" (프로덕션 확신)
```

### 3-2. 4단계 상세 + 커리큘럼 Part 매핑

```
0단계: 미니 데모 (NEW!) ─── 커리큘럼 Part 1 연동
├── 노드 7~10개, 관계 7~10개
├── 프로퍼티: ID + 이름만 (최소화)
├── 공정 3개 (혼합→열압착→연마)
├── 설비 3개 + 결함 1개 + 검사 1개
├── 핵심 질의 1개: "이 결함의 원인 공정은?"
├── 전체 파이프라인 한 바퀴:
│   스키마 → 데이터 → Neo4j 적재 → Cypher 질의 → 시각화
├── ⭐ 5분 만에 "동작하는 것"을 본다
└── 🏆 Milestone: Neo4j에서 결함→공정→설비 3-hop 시각화 확인

         ↓ "이게 되는구나!" → 더 해보고 싶다

1단계: 교육용 데모 ─── 커리큘럼 Part 2~3 연동
├── 노드 20~50개, 관계 40~60개
├── 전체 7공정 + 프로퍼티 풍부화
├── Meta-Dictionary (관계 패턴 JSON) 제작
├── 수작업 KG 구축 → LLM 자동 추출 비교
├── "구체화 > 일반화" 프롬프트 설계
├── 기본 Text2Cypher (스키마 + LLM → Cypher)
├── ⭐ 수작업 vs LLM 비교표 산출
└── 🏆 Milestone: 수작업 KG + LLM KG + 품질 비교 리포트

         ↓ "자동화해도 검수는 필수구나" → 실무 감각

2단계: 프로토타입 ─── 커리큘럼 Part 4~6 연동
├── 노드 500~1,000개
├── Entity Resolution (중복 엔티티 통합)
├── VLM 멀티모달 (표/이미지 → KG)
├── Text2Cypher Agent (generate→validate→correct→execute)
├── 하이브리드 검색 (벡터 + 그래프 + RRF)
├── Prompt Routing (Query→Task→Prompt Pool)
├── Streamlit 데모 UI
├── ⭐ 자연어로 질문하면 그래프 검색 → 답변 생성
└── 🏆 Milestone: 동작하는 GraphRAG 데모 시스템

         ↓ "진짜 작동한다!" → 서비스로 만들 수 있겠다

3단계: 서비스급 ─── 커리큘럼 Part 7 연동
├── 노드 5,000+개
├── RAGAS 평가 (Multi-hop + Common Knowledge)
├── 질문 난이도 3단계 (Easy/Medium/Hard)
├── GDBMS 선정 체크리스트 적용
├── Neo4j 성능 최적화 7가지 적용
├── 모니터링 (LangSmith/LangFuse)
├── CI/CD + Docker Compose 배포
├── 비용 최적화 (폐쇄망 vs 클라우드)
├── ⭐ 평가 지표 기반 개선 사이클 수립
└── 🏆 Milestone: 배포 가능한 GraphRAG + 품질 대시보드
```

### 5-2. 도메인별 핵심 시나리오 + 커리큘럼 연동

| 도메인 | 시나리오 | 핵심 질의 (Multi-hop) | 합의 난이도 |
|--------|---------|---------------------|-----------|
| 🏭 제조 | 브레이크 패드 결함 추적 | "접착 박리의 근본 원인 공정/설비는?" | ⭐⭐⭐⭐ |
| 💰 금융 | 거래 네트워크 + AML | "이 자금 이동 경로에 의심 패턴이 있는가?" | ⭐⭐⭐⭐⭐ |
| ⚖️ 법률 | 판례 관계 + 조항 참조 | "이 사건에 적용 가능한 유사 판례 체인은?" | ⭐⭐⭐⭐⭐ |
| 🖥️ IT | 장애 전파 + 의존성 | "DB 장애 시 영향받는 서비스 범위는?" | ⭐⭐⭐⭐ |

### 5-3. 도메인 간 재사용률

| 요소 | 재사용율 | 설명 |
|------|---------|------|
| 온톨로지 YAML 포맷 | 100% | 구조 동일, 내용만 교체 |
| 샘플 데이터 JSON 포맷 | 100% | 구조 동일, 내용만 교체 |
| Meta-Dictionary 포맷 | 90% | Prefix 9가지 공통, 도메인 키워드만 교체 |
| Neo4j 적재 스크립트 | 80% | Cypher 패턴 유사 |
| 엔티티 추출 프롬프트 | 60% | 도메인별 커스터마이징 필요 |
| Text2Cypher 기본 | 70% | 패턴 유사, 스키마만 교체 |
| Text2Cypher Agent | 50% | 그래프 패턴이 도메인마다 다름 |
| GraphRAG 파이프라인 | 70% | 라우팅 로직만 변경 |
| RAGAS 평가 코드 | 90% | 메트릭 동일, 데이터셋만 교체 |
| Streamlit 데모 | 75% | UI 뼈대 동일, 시나리오 교체 |

**핵심**: 제조 도메인을 완벽하게 끝내면, 나머지 도메인은 60~100% 복제로 빠르게 확장 가능
